# 星汐桌面伴侣 Demo 后续路线与动作规格

## 背景

当前 demo 已经具备原创 OC 桌面电子宠物/伴侣“星汐”的基础闭环：角色展示、状态机、轻触/拖拽/休息/共同学习/共同娱乐、商店与背包、投喂与赠礼、关系解锁、回忆日志、主动陪伴触发、演示触发按钮，以及 `--desktop-mode` / `--pet-mode` 桌宠模式。

本规格基于 3 个只读子评估汇总而成：

- 演示流程评估：当前 3-5 分钟演示闭环基本完整，但需要固定演示脚本和干净初始状态。
- 工程风险评估：核心边界较清晰，但逻辑时间、存档污染、demo 触发持久化和同步 LLM 风险需要先收住。
- 产品路线评估：应先强化“评委能看懂、能复现、不会被排练污染”的交付链，再扩展规则、桌宠和 LLM。

本轮又对照了以下 3 份项目文档：

- `05_Shinsekai_VPet融合开发方案.md`：强调本项目不是 Shinsekai 或 VPet 换皮，而是用 VPet 风格玩法骨架加 Shinsekai 风格 AI 表达。
- `06_产品技术Spec_AI桌面伴侣.md`：定义 MVP 必须离线可演示，按钮动作优先，AI 只能表达，状态、背包、目标和存档由本地规则决定。
- `07_美术动画与资源包工作流.md`：定义基础 9 行 atlas、57 个有效帧、候选到正式替换的 QA 边界，以及 `eat/gift/comfort/sleep` 是增强动作。

## 总控判断

下一阶段不建议先接真实 LLM，也不建议立刻继续大规模美术替换。最优先目标是把现有功能变成稳定、可复现、可讲清的 3-5 分钟演示闭环。

推荐路线是：

1. 先稳演示：演示导览、演示状态重置、存档隔离、固定口播。
2. 再扩养成：主动陪伴规则、关系/回忆可视化、触发原因解释。
3. 再收美术：保持当前 atlas 可演示，优先补 `eat` / `gift` 增强动作，不宣称发色风险已解决。
4. 再补形态：桌宠模式的拖动、停靠、返回控制面板。
5. 最后接智能：LLM 只增强表达，不改状态、不改背包、不改存档。

## P0：演示导览与存档污染治理

### 目标

让演示每次都能从同一个干净状态开始，并在 3-5 分钟内稳定展示“电子宠物/伴侣感”。

### 推荐演示路径

1. 打开控制面板模式，说明星汐是原创 OC 桌面电子宠物与伴侣，不是学习工具。
2. 点击角色区域或“轻触”，展示用户操作、角色反馈、状态变化、回忆日志。
3. 点击“共同学习”，口径限定为“共同进入专注状态”，不要描述成监督、课程提醒或效率管理。
4. 购买“热牛奶”并投喂，展示商店、背包、物品图标、状态变化和回忆。
5. 点击“模拟低能量”或“模拟久未互动”，展示星汐主动陪伴，而不是只等待用户命令。
6. 拖拽角色区域触发“拖拽/提起”。
7. 用 `--desktop-mode` / `--pet-mode` 展示桌宠形态，作为收尾画面，不把它宣称为完整桌宠系统。

### 需要实现的能力

- 演示重置：一键恢复到固定初始状态，例如 20 coins、空背包、信任 5、无回忆、非休息、初识阶段。
- 演示存档隔离：演示按钮不应污染用户正式存档，至少要支持临时演示存档、演示前重置，或 `--demo-save` / `--reset-demo-save` 一类启动参数。
- 演示导览面板：用简短步骤提示当前推荐展示动作，不写大段教程。
- 启动口径：准备一条固定命令启动普通模式，一条固定命令启动桌宠模式。
- 逻辑时间恢复：避免 controller 重启后 `now` 从 0 开始，导致久未互动、礼物衰减、主动反馈冷却出现时间倒退。
- 存档 schema：增加 `schema_version`，加载旧存档时补齐新增字段和 inventory 商品项。

### 建议任务拆分

1. `DemoStateSeed`：定义演示初始状态，和 `create_initial_state()` 保持同源但允许为演示调快关系/金币。
2. `DemoSavePolicy`：区分正式存档和演示存档，默认不让演示触发写坏正式陪伴状态。
3. `ResetDemo`：UI 或 CLI 一键重置演示状态。
4. `DemoGuide`：只显示 5-7 个短步骤，不写长教程，不遮挡角色主体。
5. `SmokeScript`：用临时 save path 跑 PySide visible smoke，避免 `data/companion_save.json` 被验证污染。

### 验收标准

- 连续运行两次演示，初始状态一致。
- 购买、投喂、主动陪伴不会因为上一次排练残留而失败。
- 关闭重开后，能明确选择继续真实存档或重置演示状态。
- 全量 `pytest` 通过。
- 如改 UI，必须重新跑 PySide visible smoke。
- `git status` 中不应因为测试或 smoke 新增默认存档副作用，除非明确记录为运行时副作用。

## P1：主动陪伴与关系/回忆可视化

### 目标

让星汐更像“有状态、有边界、会记住、会靠近”的桌面伴侣，而不是按钮反馈器。

### 推荐动作

- 增加主动陪伴触发原因展示，例如“能量偏低”“久未互动”“关系解锁后的亲近回应”。
- 将回忆日志按类型区分：互动、投喂、赠礼、使用、主动陪伴、关系解锁。
- 关系阶段可视化：初识、熟悉的陪伴、共同日常，以及下一次解锁条件。
- 增加第 3 类主动陪伴规则，但必须有冷却和不打扰边界。
- 将目标线与 06 spec 对齐：至少一个目标能在 3-5 分钟内达成或接近达成，例如“信任达到 20 解锁第一次主动称呼”。
- 将 `STAT` 和 `CHOICE` 事件尽量保持本地固定生成，让 LLM 只负责星汐的角色 speech，降低“AI 胡说状态”的风险。

### 约束

- 主动陪伴语气必须是陪伴，不是命令、监督、课程提醒或效率提醒。
- 学习/专注/休息只作为动作状态，不作为角色身份。
- 主动反馈必须继续进入 snapshot、events、memory_log 和 UI 可见区域。
- 不允许 15 秒 tick 每次刷屏。

### 验收标准

- 至少 3 类主动陪伴规则有测试覆盖。
- 连续 tick 不重复刷同类主动反馈。
- UI 能看见触发原因和最近回忆。
- 解锁反馈可在演示中稳定出现或用演示入口稳定触发。
- 关闭 LLM 后，主动陪伴与关系解锁仍完整可演示。

## P2：美术 QA 与动作资产收尾

### 目标

沿用 07 的资源工作流，在不扩大美术风险的前提下增强投喂和赠礼的视觉可读性。

### 推荐动作

- 当前基础 atlas 可用于 demo，但不能宣称所有视觉风险已解决。
- 不要把 `轻触` / `共同学习` 发色偏白说成已修复。
- 优先补 `eat` 和 `gift` 两个增强动作，因为它们直接强化背包、商店、投喂和赠礼闭环。
- `comfort` 和 `sleep` 放在 `eat/gift` 之后。
- 增强动作可以先用 Single GIF 或额外 motion 文件，不强制立即重做 8x9 atlas。
- 保持“候选 -> 可见 QA -> 正式替换 -> 重新验证 -> 人工 QA”的边界。
- 所有美术文档继续写中文。

### 验收标准

- 候选资源有几何 QA、contact sheet/GIF、PySide visible smoke 和人工 QA 记录。
- 正式替换前有回滚路径和替换记录。
- 已知搁置风险单独列出，不被当作已解决问题。
- 物品图标保持 8-12 个轻量固定商品池范围，不扩成复杂经济系统。

## P3：桌宠模式增强

### 目标

让 `--desktop-mode` / `--pet-mode` 从“形态证明”逐步变成可操作桌宠体验，但不压过控制面板中的养成闭环。

### 推荐动作

- 支持桌宠窗口拖动和停靠。
- 增加右键菜单：返回控制面板、重置演示状态、退出。
- 保持置顶和透明背景，但必须避免窗口失控。
- 后续再考虑随机游走、边缘吸附和长期驻留。

### 验收标准

- 桌宠模式可拖动、可退出、可返回控制面板。
- 5-10 分钟运行不出现空帧、窗口丢失或状态异常写坏。
- 桌宠模式仍不展示工程调试噪音。

## P4：LLM / TTS / Shinsekai 表达增强

### 总控决策

真实 LLM 不应进入下一步 P0。当前应保留规则版和 fallback，把 LLM 定位为“表达增强层”，而不是状态驱动层。TTS 可以作为增强项，但不作为核心闭环验收条件。

### 接入边界

- LLM 只生成角色台词或演出事件，每轮最多 4 个对象。
- 本地状态机决定数值、背包、关系、回忆和解锁。
- LLM 不允许修改状态、不允许创造物品、不允许写存档。
- 无 API key、超时、非法 JSON、字段越界时必须回退本地 fallback。
- PySide UI 不应被网络请求阻塞。
- `STAT` / `CHOICE` 优先由本地生成，避免模型编造状态和选项。
- 如果接 TTS，TTS 只朗读通过校验后的角色 speech，不朗读调试 JSON。

### 验收标准

- 无 LLM 时 demo 完整可跑。
- 有 LLM 时只增强表达，不改变结算结果。
- 所有 LLM 输出都经过 schema 和内容边界校验。
- 有 mock client、超时测试、fallback 测试。
- 没有 API key 时不能影响 3-5 分钟演示。

## 工程债优先级

1. 逻辑时间与存档：避免 controller 重启后 `now` 归零导致冷却、久未互动和礼物衰减异常。
2. demo 触发隔离：避免演示按钮永久污染真实陪伴状态。
3. 存档 schema：增加版本、迁移和 inventory 补全。
4. snapshot 类型化：减少 UI 与 controller 之间的松散 dict key 风险。
5. controller 拆分：后续可把主动陪伴规则、关系解锁、记忆写入、展示 DTO 分离。
6. UI 调试信息分层：公开演示界面可以压缩 JSON event 区，但保留开发者可展开查看。

## 与 05/06/07 的一致性决策

- 采用 Python/PySide 主线，不直接合并 VPet 的 C# WPF 代码。
- VPet 只作为玩法骨架参考：状态、动作、投喂、休息、动画组织和桌宠交互。
- Shinsekai 只作为表达管线参考：JSON 事件、LLM/TTS/ASR 适配思想和演出事件。
- 主角色是原创 OC 星汐；任何 UGC 样例都不能替代主角色，也不能在正式展示中模糊授权边界。
- MVP 不做复杂经济、抽卡、多人、完整 ASR、实时 T2I、MCP 外部工具。
- 美术资源不复制 VPet、Shinsekai、Vocaloid、Miku、现有 Codex pet 或其他 IP。
- 代码和文档不得把当前 demo 描述成完整商业桌宠，也不得把 LLM/TTS 描述成已完整接入，除非实际完成并验证。

## 答辩口径

- 这是原创 OC 桌面电子宠物与伴侣 demo，核心是状态、互动、养成、记忆和主动陪伴。
- “共同学习”只是动作状态，不代表角色是学习监督或效率工具。
- AI 表达层可以增强角色台词，但当前演示优先保证离线稳定和可复现。
- 当前不是完整商业桌宠系统，桌宠模式用于证明形态，后续会增强停靠、菜单和长期驻留。
- 美术资源遵守候选、QA、正式替换和人工 QA 边界，不复制既有 IP。

## 下一步推荐实施顺序

1. 写失败测试：演示重置后状态固定，背包为空，回忆为空，信任和金币回到 seed。
2. 实现演示 reset/seed，不覆盖正式存档。
3. 增加存档 schema/version 和 inventory 补全，修复重启后逻辑时间风险。
4. UI 增加“重置演示状态”或演示导览入口。
5. 跑 `pytest` 和 PySide visible smoke。
6. 再做主动陪伴触发原因可视化。
7. 最后评估是否压缩公开演示 UI 中的 JSON event 区。
